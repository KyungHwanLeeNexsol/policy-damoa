# Acceptance Criteria: SPEC-UI-001

## AC-UI-001: 정책 목록 페이지네이션

**Given** 정책 데이터베이스에 50개의 활성 정책이 있고
**When** 사용자가 `/policies` 페이지에 처음 접근하면
**Then** 20개의 정책 카드가 렌더링되고, "전체 50개 정책"이 목록 위에 표시되며, 다음 페이지 버튼이 활성화되어야 한다

**Edge case**:
- Given 5개 미만의 정책이 있을 때 → 페이지네이션 컨트롤이 표시되지 않는다
- Given `?page=3` URL 파라미터가 있을 때 → 세 번째 페이지의 결과가 표시된다

---

## AC-UI-002: 전문 검색

**Given** 제목에 "청년 월세"를 포함하는 정책 3개와 포함하지 않는 정책 17개가 있고
**When** 사용자가 검색창에 "청년 월세"를 입력하고 300ms가 경과하면
**Then** 3개의 정책이 표시되고, URL이 `?q=청년 월세`로 업데이트되며, 페이지가 1로 초기화되고, "전체 3개 정책"이 표시되어야 한다

**Edge case**:
- Given 검색어와 일치하는 정책이 없을 때 → "'청년 월세'에 대한 검색 결과가 없습니다." 메시지가 표시된다
- Given 사용자가 검색창의 X 버튼을 클릭할 때 → 검색어가 지워지고 URL에서 `?q=` 파라미터가 제거된다
- Given 입력 후 300ms 이전에 추가 입력이 있을 때 → 서버에 검색 요청이 전송되지 않는다 (디바운스)

---

## AC-UI-003: 필터 상태 URL 유지

**Given** 사용자가 `/policies?category=housing&region=11` 필터를 적용한 상태에서
**When** 사용자가 정책 카드를 클릭하여 상세 페이지로 이동한 후 브라우저 뒤로 가기 버튼을 누르면
**Then** 목록 페이지가 `category=housing&region=11` 필터가 적용된 상태로 복원되어야 한다

**Edge case**:
- Given 필터를 적용하지 않은 기본 상태에서 뒤로 가기 할 때 → 필터 없는 전체 목록이 표시된다

---

## AC-UI-004: 활성 필터 배지 표시 및 해제

**Given** 사용자가 카테고리 "주거" 및 지역 "서울특별시" 필터를 적용했을 때
**When** 정책 목록 페이지가 렌더링되면
**Then** 검색창 아래에 "주거" 배지와 "서울특별시" 배지가 각각 X 버튼과 함께 표시되어야 한다

**When** 사용자가 "주거" 배지의 X 버튼을 클릭하면
**Then** "주거" 배지가 사라지고, URL에서 `?category=housing`이 제거되며, `?region=11`은 유지되고, 새 결과가 로드되어야 한다

---

## AC-UI-005: 모바일 필터 시트

**Given** 사용자가 768px 이하 모바일 뷰포트에서 정책 목록 페이지를 보고 있을 때
**When** 사용자가 "필터" 버튼을 탭하면
**Then** 화면 하단에서 전체 높이 Sheet 컴포넌트가 올라오며 모든 필터 컨트롤(카테고리, 지역, 혜택 유형, 상태, 정렬 등)이 표시되어야 한다

**When** 사용자가 "적용" 버튼이나 Sheet 닫기 버튼을 탭하면
**Then** Sheet가 닫히고 선택된 필터가 URL에 업데이트되어 새 결과가 로드되어야 한다

---

## AC-UI-006: 로그인 사용자의 자격 체크리스트

**Given** 사용자가 UserProfile(birthYear=1995, regionId="서울특별시")이 있는 인증된 상태에서
**When** eligibilityCriteria에 나이 조건과 지역 조건이 포함된 정책 상세 페이지를 방문하면
**Then** 자격 체크리스트 섹션이 렌더링되고:
- 나이 조건이 충족되면 ✅ 녹색 체크마크와 "나이 조건: 충족" 레이블이 표시되고
- 지역 조건이 프로필과 일치하면 ✅ 체크마크가 표시되고
- 프로필에 없는 조건(예: 직업)은 🟡 노란 물음표와 "미확인" 레이블이 표시되어야 한다

---

## AC-UI-007: 비로그인 사용자의 자격 체크리스트 CTA

**Given** 인증되지 않은(비로그인) 사용자가 정책 상세 페이지를 방문했을 때
**When** 자격 체크리스트 섹션이 렌더링되면
**Then** 체크리스트 항목들이 흐릿하게(blurred) 처리되고, "로그인하면 나에게 맞는지 확인할 수 있어요" CTA 버튼이 표시되어야 한다

---

## AC-UI-008: Redis 캐시 우선 조회

**Given** 동일한 필터 조건(`?category=housing&region=11&page=1`)으로 이전에 캐시된 결과가 있을 때
**When** 해당 URL로 정책 목록을 요청하면
**Then** Prisma 데이터베이스 쿼리가 실행되지 않고 Redis 캐시에서 결과가 반환되어야 한다

**Given** Redis 서버가 응답하지 않을 때
**Then** 사용자에게 오류를 노출하지 않고 Prisma를 통해 데이터가 정상 반환되어야 한다 (graceful degradation)

---

## AC-UI-009: 빈 결과 상태 안내

**Given** 사용자가 `?category=housing&region=11` 필터를 적용했지만 일치하는 정책이 없을 때
**When** 정책 목록 페이지가 렌더링되면
**Then** "현재 조건에 맞는 정책이 없습니다"와 함께, 지역 필터를 제거하면 N개의 결과를 볼 수 있다는 구체적인 제안과 "지역 조건 제거" 원클릭 버튼이 표시되어야 한다

---

## AC-UI-010: 기존 테스트 비회귀

**Given** SPEC-INFRA-001의 110개 테스트와 SPEC-API-001의 156개 테스트가 존재할 때
**When** SPEC-UI-001 구현의 모든 마일스톤(M1~M8) 완료 후 `pnpm test`를 실행하면
**Then** 기존 266개 테스트가 모두 통과해야 한다. 새로 추가된 SPEC-UI-001 테스트도 모두 통과해야 하며, 총 테스트 커버리지는 80% 이상이어야 한다.

## Performance Acceptance Criteria

**Given** 정책이 Redis에 캐시된 상태에서
- 정책 목록 페이지 TTFB: 200ms 미만
- 정책 상세 페이지 TTFB: 150ms 미만

**Given** 캐시 미스(cold start) 상태에서
- 정책 목록 페이지 TTFB: 800ms 미만

---

## Security Acceptance Criteria

- URL 쿼리 파라미터는 서버 액션에서 검증 및 타입 변환 후 사용 (SQL 인젝션 방지)
- `page` 파라미터는 양의 정수 범위(1~MAX_PAGE_SIZE)로 제한
- `pageSize` 파라미터는 MAX_PAGE_SIZE(100) 초과 불가
- 정책 데이터는 읽기 전용; 사용자 입력으로 정책 데이터 수정 불가
